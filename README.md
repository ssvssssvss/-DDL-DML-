Домашнее задание к занятию «Репликация и масштабирование. Часть 2». Грекова Иоланта.

**Задание 1**
Опишите основные преимущества использования масштабирования методами:
активный master-сервер и пассивный репликационный slave-сервер;
master-сервер и несколько slave-серверов;
Дайте ответ в свободной форме.

**Ответ**
Активный master и пассивный slave:
- Повышение отказоустойчивости: 
- Все операции записи выполняются на master-сервере, а slave хранит его копию и может быть быстро переведен в рабочий режим при сбое. 
- Можно использовать slave для чтения, снижая нагрузку на основной сервер.
Master и несколько slave-серверов:
- Позволяет эффективно масштабировать систему по чтению. 
- Запросы на чтение распределяются между несколькими slave-серверами, что увеличивает производительность.
- Повышается отказоустойчивость, так как при выходе из строя одного slave остальные продолжают работать.
- Несколько slave → масштабирование чтения + высокая доступность

**Задание 2**
Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:
пользователи,
книги,
магазины (столбцы произвольно).
Опишите принципы построения системы и их разграничение или разбивку между базами данных.
Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.

**Ответ**
Принцип пработы:
1. Имеется 3 таблицы: users, books, shops.
2. Нужно масштабировать: разделить по функционалу (вертикально) и по данным (горизонтально).

Вертикальный шардинг:
1. По разным базам/серверам.
- Users DB → всё про пользователей
- Books DB → каталог книг
- Shops DB → магазины и наличие товаров
2. Разделение:
Вот данные, оформленные в формате Markdown-таблицы:

| База данных | Таблицы |
|-------------|---------|
| DB1         | users   |
| DB2         | books   |
| DB3         | shops   |

3. Причины разделения:
- снижение нагрузки
- изоляция данных
- упрощение масштабирования

Горизонтальный шардинг (по данным): теперь внутри каждой БД разделить данные.
1. Users (по user_id)
shard1 → user_id 1–1M
shard2 → user_id 1M–2M
shard3 → и т.д.
2. Books (по жанру)
shard1 → fiction
shard2 → science
shard3 → education
3. Shops (по регионам)
shard1 → Москва
shard2 → Рязань
shard3 → Казань

Принцип маршрутизации: добавляется роутер, который определяет, в какой шард отправить запрос:
- регион
- жанр
- user_id

Репликация (режимы работы серверов):
Для каждого шарда: Master-Slave
Master → запись (INSERT/UPDATE)
Slave → чтение (SELECT)
- масштабирование чтения
- отказоустойчивость

Master-Master (опционально):
- для критичных данных (users)
- оба сервера принимают запись
- высокая доступность
- сложность (конфликты)

Принцип работы: клиент делает запрос, который поступает на API. API отправляет в router, который определяет shard (горизонтальный) и определяет нужную БД (вертикальный шардинг):
- в slave (если чтение)
- в master (если запись)
