**Задание 1. СУБД**
*Кейс*
*Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.*
*Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?*
*1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.*
*1.1. Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?*
*1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.*
*1.2. Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?*
*1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.*
*1.3. Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?*
*1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.*
*1.4. Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?*
*1.5. Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?*
*Приведите ответ в свободной форме.*

**Ответ:**
1.1 Бюджетирование и финансовая аналитика.
Для задач, связанных с бюджетированием, финансовой отчётностью и прогнозированием рисков, оптимально использовать реляционные СУБД (SQL), такие как PostgreSQL или Microsoft SQL Server:
- поддержка транзакций (ACID), что критично для финансовых данных;
- строгая структура данных и схема;
- высокая целостность данных;
- развитые возможности аналитических запросов (SQL).

1.1 Ускорение хеширования.
Если операции хеширования выполняются слишком долго, следует использовать специализированные криптографические библиотеки с аппаратным ускорением, например:
- OpenSSL API;
- libsodium.
Они задействуют аппаратные инструкции процессора (например, AES-NI), а это ускоряет вычисления по сравнению с программной реализацией.

1.2 Лендинги и CRM
Для обработки данных лендингов и CRM лучше использовать NoSQL СУБД, такие как MongoDB или Redis.

Лендинги имеют большое количество быстрых операций записи и простую/слабо структурированная модель данных. Подходит Redis (как кэш) или MongoDB.
CRM имеют гибкую структуру данных и частые изменения схемы. Оптимально использовать MongoDB (документо-ориентированная модель).

1.2 Можно ли использовать одну СУБД?
Да, можно использовать одну СУБД — например, MongoDB.

Она может:
- хранить данные в формате JSON-документов;
- гибко изменять структуру;
- слабее поддержка транзакций.

1.3 База знаний (нормы, обучение).
Для хранения корпоративных норм, регламентов и обучающих материалов подойдут:
- реляционные СУБД (например, PostgreSQL);
- документо-ориентированные СУБД (например, MongoDB).
- если структура фиксированная → SQL;
- если требуется гибкость → NoSQL.

1.3 Использование существующей СУБД
Да, можно использовать уже внедрённую СУБД, например MongoDB.
Реализация:
1. выделение отдельной коллекции (например, knowledge_base);
2. хранение документов с иерархией (категории, статьи, вложенность).

1.4 Логистика и маршруты

Для задач построения маршрутов и работы со связями оптимально использовать графовые СУБД, например Neo4j:
- эффективное хранение связей между объектами;
- быстрый поиск маршрутов (алгоритмы графов);
- удобство моделирования логистических процессов.

1.4 Интеграция с отделом закупок
Для отдела закупок лучше использовать отдельную реляционную СУБД (например, PostgreSQL):
- данные имеют табличную структуру (контракты, поставщики);
- важна целостность данных.
- интегрирация её с графовой СУБД логистики через API.

1.5 Можно ли использовать одну СУБД для всех задач?
Возможно (например, использовать только PostgreSQL или MongoDB), но на практике это неэффективно:
- разные типы задач требуют разных моделей данных;
- ухудшается производительность;
- усложняется масштабирование.

Лучший подход — использование нескольких СУБД:
Финансы → PostgreSQL
CRM → MongoDB
Лендинги → Redis / MongoDB
База знаний → MongoDB или PostgreSQL
Логистика → Neo4j


**Задание 2. Транзакции**
*2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.*
*2.1. Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?*
*Приведите ответ в свободной форме.*

**Ответ:**
2.1 Пополнение баланса телефона
При пополнении баланса телефона выполняется транзакция — последовательность действий, которая должна завершиться полностью или не выполниться вовсе.
1. Инициация транзакции: Пользователь вводит номер телефона и сумму пополнения, система создаёт новую транзакцию в базе данных.
2. Проверка данных. Система проверяет: корректность номера телефона и достаточность средств на счёте пользователя (или платёжного метода).
3. Блокировка средств: На стороне платёжной системы происходит резервирование (холд) необходимой суммы, чтобы избежать двойного списания.
4. Списание средств: После успешной проверки средства списываются со счёта пользователя.
5. Зачисление средств: Деньги зачисляются на баланс мобильного оператора (или увеличивается баланс телефона в системе).
6. Фиксация транзакции. Транзакция подтверждается: изменения записываются в базу данных и пользователь получает уведомление об успешной операции.
Если на любом этапе возникает ошибка: выполняется откат транзакции и средства возвращаются пользователю. Система остаётся в согласованном состоянии.

2.1* Пополнение через автоплатёж
Автоплатёж — это та же транзакция с автоматическим запуском.
Отличия:
1. Срабатывание триггера: система автоматически инициирует платёж при выполнении условия: достижение минимального баланса и наступление даты платежа.
2. Проверка привязанного метода оплаты: валидность карты или счёта и доступность средств.
3. Автоматическая инициация транзакции: создаётся транзакция без участия пользователя.
4. Резервирование и списание средств.
5. Зачисление средств на баланс: баланс телефона пополняется.
6. Завершение транзакции и уведомление. Транзакция фиксируется, пользователю отправляется уведомление (например, push или SMS).
Ключевое отличие автоплатежа: нет ручного ввода данных и используется заранее сохранённая платёжная информация + добавляется триггер (условие запуска).

**Задание 3. SQL vs NoSQL**
*3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.*
*3.1. Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.*
*Приведите ответ в свободной форме*

**Ответ:**
3.1 Преимущества SQL-систем над NoSQL
1. Поддержка транзакций (ACID). SQL-системы обеспечивают:
- атомарность;
- согласованность;
- изолированность;
- надёжность.
Это важно для финансовых операций и критичных данных.
2. Строгая схема данных:
- чётко определённая структура таблиц;
- контроль типов данных;
- предотвращение некорректных записей.
В NoSQL схема часто отсутствует или гибкая, что может приводить к ошибкам.
3. Поддержка сложных запросов:
-мощный язык SQL (JOIN, агрегаты, подзапросы);
-удобство аналитики и построения отчётов.
-В NoSQL такие операции либо сложнее, либо менее эффективны.
4. Обеспечение целостности данных:
- внешние ключи (FOREIGN KEY);
- ограничения (constraints);
- триггеры.
-  Это снижает риск несогласованных данных.
5. Зрелость и стандартизация
- SQL — стандарт;
- большое количество инструментов и специалистов;
- стабильность и предсказуемость поведения.
3.1* Преимущества NewSQL над SQL и NoSQL
Преимущества NewSQL:
1. Горизонтальное масштабирование
- масштабируются как NoSQL;
- можно добавлять новые узлы без остановки системы.
- В классических SQL это ограничено.
2. Поддержка ACID при масштабированииЖ
- сохраняют транзакции даже в распределённой среде;
- обеспечивают согласованность данных.
- В NoSQL часто используется eventual consistency.
3. SQL-интерфейс:
- используют привычный SQL;
- упрощают миграцию с классических СУБД.
4. Высокая отказоустойчивость
- репликация данных между узлами;
- автоматическое восстановление при сбоях.
5. Баланс между производительностью и консистентность.
- быстрее классических SQL в распределённых системах;
- более консистентны, чем NoSQL.

**Задание 4. Кластеры**
*Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.*
*На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?*
*Приведите ответ в свободной форме.*

**Ответ:**
При обработке большого объёма данных с использованием 1000 машин ключевой критерий выбора СУБД - способность к горизонтальному масштабированию и работе в распределённой среде.
Важно учитывать:
- тип нагрузки (в данном случае — аналитическая, OLAP);
- возможность распределённого хранения данных;
- отказоустойчивость системы.
Для таких задач подходят распределённые решения (Apache Hadoop, Apache Cassandra).
Модель распределённых вычислений:
- Самая подходящая модель MapReduce:
- разбивает задачи на параллельные подзадачи;
- эффективно обрабатывает большие объёмы данных;
- обеспечивает отказоустойчивость;
- масштабируется на большое количество узлов.
Еще один вариант вариант — Apache Spark, который выполняет вычисления быстрее за счёт работы в памяти и лучше подходит для сложной аналитики.
Для данной задачи оптимально использовать распределённую СУБД и модель MapReduce или Apache Spark, так как они обеспечивают масштабируемость, параллельную обработку и устойчивость к отказам в кластере большого размера.
