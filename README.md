**Задание 1. Резервное копирование**
Кейс
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.
Необходимо описать, какие варианты резервного копирования подходят в случаях:
1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.
1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.
1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.
Приведите ответ в свободной форме.

**Ответ**
1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день:
Полное резервное копирование, выполняемое ежедневно. 
Альтернатива: Full + Differential, где полная копия делается реже (например, раз в неделю), а ежедневно — дифференциальная, содержащая изменения с момента последней полной.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.
Это восстановление на момент времени (Point-in-Time Recovery). Подойдет резервное копирование журналов транзакций с интервалом 15–60 минут: позволяет откатить базу к любому заданному моменту. 
Еще может подойти инкрементное копирование с часовым интервалом.

1.3. Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных?
Да, такой кейс реализуется средствами высокой доступности:
- Отказоустойчивый кластер — при выходе одного сервера из строя другой автоматически подхватывает работу.
- Синхронная репликация — данные одновременно пишутся на основной и резервный серверы, переключение происходит автоматически за секунды.


**Задание 2. PostgreSQL**
2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).
2.1. Возможно ли автоматизировать этот процесс? Если да, то как?
Приведите ответ в свободной форме.

**Ответ**
2.1. Примеры команд резервирования и восстановления БД
Резервное копирование (pg_dump):
# SQL-файл (универсальный формат)
pg_dump mydatabase > mydatabase.sql
# Custom-формат (сжатый, рекомендуется для больших БД)
pg_dump -Fc mydatabase > mydatabase.dump

Восстановление (pg_restore / psql):
# Из SQL-файла
psql -d mydatabase < mydatabase.sql
# Из custom-формата
pg_restore -d mydatabase mydatabase.dump

2.1. Автоматизация процесса
Да, процесс автоматизируется. Возможно несколько способов:
1. Скрипт с ротацией — создаётся bash-скрипт, который выполняет резервирование, добавляет дату в имя файла и удаляет устаревшие копии:
#!/bin/bash
BACKUP_DIR="/backups"
DATE=$(date +"%Y%m%d_%H%M%S")
pg_dump -U postgres mydb | gzip > "$BACKUP_DIR/mydb_$DATE.sql.gz"
find $BACKUP_DIR -name "*.sql.gz" -mtime +7 -delete

2. Cron — для регулярного выполнения:
0 2 * * * /usr/local/bin/backup_postgres.sh
3. Готовые инструменты — autopostgresqlbackup. В нем есть встроенные механизмы сжатия, ротации и уведомлений.

**Задание 3. MySQL**
3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.
3.1. В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?
Приведите ответ в свободной форме.

**Ответ**
3.1. Пример команды инкрементного резервного копирования MySQL

Для подобных целей используется утилита mysqlbackup:
# Полное резервное копирование (база для инкрементов)
mysqlbackup --user=root --password=password --backup-dir=/backups/full backup
# Инкрементное резервное копирование
mysqlbackup --user=root --password=password --backup-dir=/backups/incr \
  --incremental --incremental-base=dir:/backups/full backup
Разбор синтаксиса:
- --incremental — указывает на создание инкрементной копии
- --incremental-base=dir:путь — задаёт базовую директорию с предыдущей копией (полной или инкрементной)
- --start-lsn=LSN — альтернативный способ указания начальной Log Sequence Number

3.1. Преимущества репликации перед обычным резервным копированием
1. Горизонтальное масштабирование чтения — запросы на чтение (SELECT) можно распределять между несколькими репликами, снижая нагрузку на основной сервер. Эффективно, когда чтение преобладает над записью.
2. Высокая доступность — в случае отказа основного сервера можно оперативно переключиться на реплику, минимизируя время простоя. Реплика выступает как "горячий" резерв.
3. Выгрузка backup-нагрузки — резервное копирование выполняется на реплике, а не на основном сервере. Это исключает влияние backup-процесса на продуктивную рабочую нагрузку.
4. Выполнение аналитических запросов — тяжёлые аналитические задачи и отчёты можно направлять на отдельные реплики, не замедляя операционную базу.
